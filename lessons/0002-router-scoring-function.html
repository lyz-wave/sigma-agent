Router 精排打分函数

Lesson 002 · 阅读时间 8 分钟 · ← Lesson 001

上一课说了框架的全貌。这一课深入最微妙的一个设计——Stage 2 精排的打分函数。

为什么要单独讲这个？因为它是整个项目设计哲学的缩影：在正确的环节用正确的手段。路由需要精度，但不是所有精度都需要 LLM。

核心公式

打分函数是一个加权线性组合，三个维度各自贡献一部分分数：

score = wi · si + wt · st + we · se
0.5
意图权重
意图精确匹配则满分，否则 0
这是主导维度
0.3
标签权重
查询标签中有多大比例命中了 Skill 的标签集
0.2
示例权重
用户查询文本与 Skill 示例的关键词重叠度

三个权重的和为 1.0，保证输出在 [0, 1] 区间内。

为什么不用 LLM 做精排？

这是本项目最重要的设计决策之一。如果你让 LLM 来做 Stage 2 的最终选择：

维度打分函数LLM 精排
Token 成本 零 — 纯数学运算 每次路由消耗 N 个 Skill × 若干 Token
延迟 微秒级 百毫秒到秒级
确定性 确定 — 相同输入始终相同输出 随机性 — 同样的输入可能选不同 Skill
路由本身引入的噪声 无 LLM 的选择偏差可能抵消路由的收益

设计哲学：如果路由本身也需要 LLM 来决策，那路由带来的 Token 节省就完全被它自己消耗掉了。打分函数的存在是为了避免在路由过程中调用 LLM，而不是作为 LLM 的补充。

逐项拆解

1. 意图匹配分（权重 0.5）

最直接的维度。用户查询被分类为一个 intent_class，Skill 注册时也声明了自己的 intent_class。

# 精确匹配 → 满分
intent_score = 1.0  （当 candidate.intent_matched == True）

# 不匹配 → 零分
intent_score = 0.0  （否则）

这是硬约束：如果意图都不匹配，这一项就是 0。但因为权重 0.5，标签和示例仍有总计 0.5 的权重，所以其他维度可能"救"回来——这是有意为之，用于处理模糊查询。

2. 标签重叠分（权重 0.3）

用户查询中提取的标签，有多大比例命中了 Skill 的标签列表。

# 例子：查询标签 ["python", "quality", "deploy"]
skill_tags = ["code", "review", "quality", "python"]
matched_tags = ["python", "quality"]

tag_score = 2/3 ≈ 0.667

3. 示例相似度（权重 0.2）

将用户的原始查询文本与 Skill 注册时提供的 examples 列表做关键词重叠。

# 用户查询："review this PR for bugs"
query_words = {"review", "this", "pr", "for", "bugs"}
→ 去掉短词后: {"review", "this", "pr", "bugs"}

# Skill 示例："review this PR for bugs", "check code style"
example_words = {"review", "this", "pr", "for", "bugs", "check", "code", "style"}

common = {"review", "this", "pr", "bugs"}

example_score = 4/4 = 1.0

完整计算示例

假设我们有三个 Skill 注册在 Catalog 中：

Skillintent_classtagsexamples
code-reviewcode_review["code","review","quality","python"]"review this PR for bugs"
test-runnertesting["test","python","quality"]"run all tests"
deploydeployment["deploy","devops","infra"]"deploy to production"

现在用户的查询是 "review the test quality"，提取为 intent_class=""（不确定）、tags=["review", "test", "quality"]：

Stage 1 召回：
code-review ← 命中 tags: "review", "quality"
test-runner ← 命中 tags: "test", "quality"
deploy ← 零匹配 — 排除

Stage 2 评分：

code-review
intent_score = 0.0 (intent_class 为空，无意图匹配)
tag_score = 2/3 = 0.667
example_score: query="review the test quality" vs "review this PR for bugs"
→ common={"review"} → 1/4 = 0.25
→ 总分 = 0.5×0 + 0.3×0.667 + 0.2×0.25 = 0.25

test-runner
intent_score = 0.0
tag_score = 2/3 = 0.667
example_score: "review the test quality" vs "run all tests"
→ common={"test"} → 1/4 = 0.25
→ 总分 = 0.5×0 + 0.3×0.667 + 0.2×0.25 = 0.25

平局！当规则不够区分时——这就是为什么 Stage 2 之后还有 Orchestrator 做最终决策。

可配置性

权重不是写死的。构造 Router 时可以自定义：

# 如果你更依赖标签匹配而非意图（比如意图分类不够准）
router = Router(
    registry,
    intent_weight=0.3,
    tag_weight=0.5,
    example_weight=0.2,
)

# 或者完全去掉示例相似度
router = Router(
    registry,
    intent_weight=0.6,
    tag_weight=0.4,
    example_weight=0.0,
)

验证：没有 LLM 调用

框架中有一个测试专门验证精排阶段没有 LLM 调用：

def test_no_llm_at_rank_time(self, router):
    import inspect
    source = inspect.getsource(router._score)
    assert "openai" not in source.lower()
    assert "anthropic" not in source.lower()
    assert ".chat" not in source.lower()

它检查的是源代码中是否包含常见 LLM 库的关键词——如果未来有人不小心在打分函数里加了 LLM 调用，这个测试会失败。

总结

要记住的一句话：打分函数不是"穷人的 LLM"，而是有意为之的设计选择——路由的决策需要快速、确定、低成本，这些恰好是 LLM 不擅长的。

这也引出了一个更通用的问题框架：在你的系统里，哪个环节必须用 LLM，哪个环节用规则/数学就够用了？ 这个区分往往是好的架构和差的设计之间的分水岭。
