Sigma Agent 全貌

Lesson 001 · 阅读时间 10 分钟 · 对应 Mission

Sigma Agent 是一个 AI Coding Agent 框架，核心思路是：以 Query Loop + Tool Use 构建任务执行闭环。它解决的是真实世界中 Agent 框架的三个核心问题：

  1. 检索噪声与功能重叠 — Tool 多了以后 Agent 选不对、选不准
  2. 经验无法跨会话复用 — 每次启动从零开始，同样的错反复犯
  3. 长会话上下文退化 — 跑久了关键信息被稀释，推理准确率下降
💡 讲解要点：这是项目的"电梯演讲"。用这三句话能让听众在 10 秒内理解项目价值。

系统架构

框架由 6 个模块构成，按执行顺序串联：

用户输入
↓
Router — 两阶段（召回 + 精排）→ 选出最佳 Skill
└─ Stage 1: 意图分类 + 标签匹配
└─ Stage 2: 打分函数（无 LLM 调用）
↓
Skill Executor — 执行 Tool 编排序列
└─ 参数传递 + 错误传播 + 失败跳过后续步骤
↓
ReflectionEngine — 反思执行结果
└─ 成功 → 提取 ProceduralMemory
└─ 失败 → EpisodicMemory + 修复步骤
↓
Memory System — 分类存储 · 索引更新 · 按需复用
↓
Context Compressor — 四阶段逐级压缩（OR 触发）
↓
输出 / 下一轮迭代

四大技术支柱

1. Skill 分层路由

解决的问题：Tool 数量从 10 个增长到 100 个后，Agent 无法准确选择合适的能力单元。

方案：三层组织结构 + 两阶段路由。

Tool (原子操作) → Skill (多 Tool 编排) → Skill Catalog (注册中心)

路由过程：

  • Stage 1 — 召回：基于意图分类 + 元信息标签匹配，从 Catalog 中筛选候选 Skill 集合
  • Stage 2 — 精排：用打分函数（加权组合：意图匹配 0.5 + 标签重叠 0.3 + 示例相似度 0.2）从候选中选最优
关键设计决策：精排用打分函数而非 LLM。
原因：精排阶段如果调 LLM 做选择，路由本身的 Token 成本就抵消了它意图解决的问题。打分函数是纯数学运算，确定性强、成本可预测。

2. 自进化记忆沉淀

解决的问题：Agent 每次启动从零开始，同样的错误无法避免。

方案：执行后自动反思，提炼经验存入三种记忆类型。

类型内容来源
FactualMemoryAPI 参数、key=value 配置等静态知识成功执行中自动提取
ProceduralMemoryHow-to 步骤、正确模式成功执行 → 提取步骤
EpisodicMemory失败归因 + 上下文失败执行 → 错误 + 环境

生命周期闭环：执行 → 反思 → 提炼 → 分类存储 → 索引更新 → 按需复用

双路径策略：成功时提取正样本（程序性经验），失败时提取负样本（错误归因 + 修复步骤），确保两种经验都有沉淀。

3. 分层上下文压缩

解决的问题：长会话中关键上下文被稀释，Agent 推理准确率衰减。

方案：四阶段逐级执行 pipeline。

摘要预览 → 占位替换 → 按需检索 → 超限兜底
  • Stage 1：将大工具输出压缩为结构化摘要
  • Stage 2：用缓存友好型占位符替换大内容块
  • Stage 3：需要时从占位符恢复完整内容
  • Stage 4：超出硬限制时裁剪，保留最近 N 条关键项

触发条件（OR）：Agent 自省 / 轮数阈值 / Token 阈值。

4. 中心化多 Agent 协作

解决的问题：复杂任务需要分解并行执行，但 Agent 间协调基础设施过于复杂。

方案：主 Agent 统一规划审批，子 Agent 以 Tool Call 方式受控执行。

主 Agent 制定计划
  ↓ (Tool Call 调度)
子 Agent 接收到目标
  ↓ (自规划执行)
子 Agent 调用 Tool → 执行步骤
  ↓ (返回结果)
主 Agent 评估、审批、下一步决策

子 Agent 约束：

  • 全量 Tool 访问权限
  • 用完即销毁（无状态跨步骤复用）
  • 只读记忆（可查询经验，不可写入）
  • 支持 Fork（进程内并行）和 Worktree（隔离工作区）模式
关键设计决策：避免引入复杂的 Agent 间协调基础设施。
所有协调通过主 Agent 完成，没有子 Agent 之间的直接通信。这大幅降低了系统复杂度，代价是主 Agent 成为瓶颈——但对于框架类项目，这种折衷是可接受的。

讲解话术建议

Q: 这个项目和其他 Agent 框架有什么区别？
大多数框架止步于 Tool Calling 层面——给你一堆工具让 Agent 自己选。Sigma Agent 加了组织层次（Skill）、经验沉淀（Memory）、上下文管理（Compressor）和协作模式（Orchestrator），让 Agent 在长期运行中越用越聪明，而不是越用越乱。
Q: 为什么叫 Sigma？
Σ 是数学求和符号，代表"汇聚"——将多个工具汇聚为 Skill，将多次经验汇聚为记忆，将多个 Agent 汇聚为协作。
Q: 生产可用吗？
MVP 阶段——核心架构完整，81 个测试通过。路由、记忆、压缩、协作四个闭环都已实现。下一步可以加 CLI 入口、真实 Skill 示例和 PyPI 发布。
