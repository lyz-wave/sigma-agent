用户输入
→
意图分类
→
标签提取
→

候选 Skill 集合

目前项目中，前面两步（意图分类 + 标签提取）还没有实现，Router 直接接收已经提取好的 intent_class 和 tags。我来解释每步的设计意图和实现思路。

两层过滤的逻辑

Stage 1 的核心目标是：快速排除不相关的 Skill，缩小 Stage 2 精排需要处理的候选集。它有两个过滤层，一宽一窄：

第一层：意图分类

将用户的自然语言查询映射到预定义的意图类别。这是最粗的过滤——如果你的 Skill Catalog 有 50 个 Skill，可能有 10 个意图类别。意图分类的一步就能排除 80% 的不相关 Skill。

如何实现？ 这里应该用一个小 LLM 调用或关键词分类器。

关键点：Stage 1 是唯一适合调 LLM 的路由环节——因为它一次调用就能排除大量候选，后续的 Stage 2（打分函数）和整个执行流程都不再需要 LLM。这笔 Token 花得值。

一个简单的实现思路：

INTENT_KEYWORDS = {
    "code_review": ["review", "pr", "code quality", "lint", "check style"],
    "testing":     ["test", "unit test", "integration test", "pytest"],
    "deployment":  ["deploy", "release", "push", "rollback"],
    "refactoring": ["refactor", "restructure", "clean up", "technical debt"],
}

def classify_intent(query: str) -> str:
    q = query.lower()
    scores = {}
    for intent, keywords in INTENT_KEYWORDS.items():
        scores[intent] = sum(1 for kw in keywords if kw in q)
    return max(scores, key=scores.get) if max(scores.values()) > 0 else "unknown"

第二层：标签匹配

进一步的细化过滤。从用户查询中提取技术标签（如 "python"、"docker"、"api"），与 Skill 注册时声明的标签列表做交集匹配。

标签的设计目的是捕捉意图之外的上下文——同样是 "deploy"，"deploy 到 kubernetes" 和 "deploy 到 serverless" 的标签截然不同。

def extract_tags(query: str, known_tags: set[str]) -> list[str]:
    q_words = set(query.lower().split())
    return [t for t in known_tags if t in q_words]

intent_class = "testing"
# 精确匹配 intent_class
for skill in self._skills:

tags = ["python", "api"]

两条召回路径

看 registry.query() 的源码，它实际有两条路径：

路径条件召回逻辑
意图路径 intent_class 不为空 只召回 intent_class 精确匹配的 Skill
skill.metadata.intent_class == intent_class
标签路径 intent_class 为空
但 tags 不为空		 召回任意标签重叠的 Skill
any(t in skill.tags for t in tags)

设计意图：

  • 意图路径：确定性高，只返回精确匹配的 Skill，候选项精而少
  • 标签路径：兜底策略——当意图分类不确定时（如用户输入太短或模糊），退而求其次用标签做更宽泛的匹配

完整流程串起来

用户输入：  "review this PR and check for style issues"
            ↓
意图分类：  intent_class = "code_review"    ← 关键词 "review" 匹配
标签提取：  tags = ["python", "style"]      ← 假设已知标签库含这些
            ↓
Registry.query(intent_class="code_review", tags=["python", "style"])
            ↓
召回结果：  [code-review]                   ← 只有一个 Skill 匹配
            ↓
Stage 2 精排看 Lesson 002                     ← 对候选打分排序

为什么框架没实现这部分？

因为意图分类的实现方式高度依赖于使用场景：

  • 如果你在 Claude Code 中使用，可以用 /skill 或关键词路由
  • 如果你的系统自己有 NLP pipeline，可以直接用 embedding 分类
  • 如果只是为了快速演示，一个关键词字典就够用了

框架把这部分作为接口暴露（传入 intent_class + tags），让使用者根据自己场景选择最合适的实现，而不是硬编码一种分类方式。

要记住的一句话：Stage 1 的召回是"快滤"不是"精排"——它的目标是用最小的成本排除明显不相关的 Skill，把最难的选择留给 Stage 2 的打分函数。快滤可以粗一点，但必须快。
