Orchestrator：中心化多 Agent 协作

Lesson 008 · 阅读时间 8 分钟 · ← Skill Executor

Orchestrator 是框架中最上层的模块——它负责把复杂任务分解、分配给 Worker、评估结果、汇总输出。

架构

🧑‍💼 Orchestrator
计划 · 调度 · 评估 · 汇总
↓ plan() 分解任务 ↓
👷 Worker A
build → test
👷 Worker B
deploy → notify
↑ evaluate() 评估结果 ↑
📊 aggregate()
最终输出

完整生命周期

1. plan("Build, test, and deploy")
   ↓ 识别关键动作，按依赖排序
   subtasks = [
     build (depends_on: []),
     test  (depends_on: [build]),
     deploy(depends_on: [test]),
   ]

2. execute_plan(plan, mode=SEQUENTIAL)
   ↓ 按依赖拓扑依次或并行执行
   for subtask in plan:
     create Worker → execute → destroy Worker

3. evaluate(result)
   ↓ 判断是否接受
   accepted = result.success

4. aggregate(results)
   ↓ 合并输出
   "[build] compiled\n[test] 42 passed\n[deploy] deployed to staging"

Worker 的生命周期

阶段行为
创建Orchestrator 为每个 subtask 创建一个新的 Worker 实例，注入全量 Tool 和只读记忆句柄
执行Worker 在 subtask 边界内自行规划执行顺序，调用所需 Tool
销毁执行完成后 Worker 对象被丢弃，不维护任何状态

为什么用完就销毁？

  • 防止状态泄漏——Worker 之间不共享状态，一个 Worker 的变量不会意外影响另一个 Worker
  • 简化心智模型——每个 Worker 只处理一个 subtask，不需要考虑"上次调用的上下文"
  • 与分布式兼容——无状态 Worker 可以轻松迁移到远程执行

三种执行模式

模式含义适用场景
SEQUENTIAL 串行执行 subtask，按依赖拓扑排序
依赖不满足则标记为 skipped		 有明确依赖链的任务（build → test → deploy）
CONCURRENT 并行执行无依赖的 subtask
按依赖分批提交到 ThreadPoolExecutor		 多个独立子任务可同时进行
WORKTREE 每个 subtask 在独立临时目录中执行
执行完自动清理目录		 子任务会产生文件/目录冲突的场景

依赖拓扑是如何被执行的？

plan.subtasks = [
  Subtask(id="build",  depends_on=[]),
  Subtask(id="test",   depends_on=["build"]),
  Subtask(id="deploy", depends_on=["test"]),
  Subtask(id="notify", depends_on=[]),          # 可并行
]

SEQUENTIAL 模式下：
  build (无依赖)       → 成功
  test (依赖 build)    → build 成功了，可以执行
  deploy (依赖 test)   → test 成功了，可以执行
  notify (无依赖)      → 可以执行（与 deploy 并行无关）

CONCURRENT 模式下：
  第 1 批：build, notify  ← 两个都无依赖，同时提交
  第 2 批：test           ← 等待 build 完成
  第 3 批：deploy         ← 等待 test 完成

依赖不满足时

如果 build 失败，依赖链会传播：
  build  → fail ❌
  test   → skipped (依赖 build 失败)
  deploy → skipped (依赖 test 未完成)

最终的 aggregate() 输出：
  [build] FAILED: build error
  [test] SKIPPED: dependency failed (build)
  [deploy] SKIPPED: dependency failed (test)
  ---
  3 subtask(s) failed

设计决策：为什么中心化？

框架选择「主 Agent 统一规划审批」而非「Agent 间自由协商」，原因：

  • 避免协调基础设施——Agent 间通信、仲裁、共识这些基础设施往往比它解决的问题还复杂
  • 主 Agent 是瓶颈，但可接受——在框架层面，自然不会有上万个 Worker 并发执行；如果是几十个 Worker 级别，中心化完全够用
  • 关注点分离——Orchestrator 做决策（做什么），Worker 做执行（怎么做）
关键设计决策：子 Agent 之间没有直接通信。所有协调通过 Orchestrator 完成。这意味着一百个 Worker 之间不会出现死锁或消息风暴——但代价是 Orchestrator 成为单点瓶颈。对于框架层项目，这个折衷是合理的。

要记住的一句话：Orchestrator 做的不是「让 Agent 协作」，而是「不要让 Agent 因为协作而互相干扰」。Worker 是纯粹的「执行单元」，不需要知道其他 Worker 的存在。
