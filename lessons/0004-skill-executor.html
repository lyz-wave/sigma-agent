Skill Executor：Tool 编排执行

Lesson 004 · 阅读时间 6 分钟 · ← Stage 1 召回

Router 选出了最佳的 Skill 之后，下一步就是执行它。这就是 SkillExecutor 的职责。

Skill 是什么？

先回顾 Skill 的定义：

@dataclass
class Skill:
    id: str
    name: str
    description: str
    metadata: dict          # tags, intent_class, applicability_boundary
    examples: list[str]
    tools: list[ToolRef]    # 这个 Skill 涉及哪些 Tool
    pipeline: list[ExecutionStep]  # 执行步骤编排

Skill = 多个 Tool 的编排序列，不是 Tool 的 wrapper。关键在 pipeline 字段——它定义了工具的执行顺序和参数传递。

执行逻辑

SkillExecutor.run() 的代码极其精简：

def run(self, skill: Skill) -> ExecutionResult:
    outputs = []
    for step in skill.pipeline:
        tool = self._tools.get(step.tool)
        if tool is None:                         # ← 工具不存在

            return ExecutionResult(success=False, error=f"Tool '{step.tool}' not found")
        result = tool.execute(step.params)       # ← 执行当前步骤
        outputs.append(f"{step.tool}: {result.output}")
        if not result.success:                   # ← 失败则跳过后续步骤
            return ExecutionResult(success=False, outputs=outputs, error=result.error)
    return ExecutionResult(success=True, outputs=outputs)

逐行拆解：

for step in skill.pipeline:
遍历 Skill 定义中的执行步骤序列
tool = self._tools.get(step.tool)
从已注册的工具字典中查找对应的 Tool 实例。如果找不到，立即返回错误。
result = tool.execute(step.params)
执行工具，传入该步骤的参数。每个 Tool 都实现同一个接口：execute(params) → ToolResult
outputs.append(...)
记录执行输出，如 "tool_a: result data"
if not result.success: return error
如果当前步骤失败，返回已完成的输出 + 错误信息，跳过后续所有步骤

错误传播机制

看具体的失败场景：

pipeline = [
    ExecutionStep(tool="build", params={"target": "prod"}),
    ExecutionStep(tool="deploy", params={"env": "staging"}),
    ExecutionStep(tool="notify", params={"channel": "slack"}),
]
  • 如果 build 失败 → 返回 ExecutionResult(success=False, outputs=["build: ..."], error="build failed") — deploy 和 notify 被跳过
  • 如果 deploy 失败 → outputs=["build: ...", "deploy: ..."]，notify 被跳过
  • 如果 Tool 未注册 → 立即返回错误，不会尝试执行
设计选择：为什么失败后跳过后续而不是重试？因为重试逻辑是 Orchestrator 的职责，SkillExecutor 只负责"按顺序执行一次"。这是关注点分离——如果每层都做重试，逻辑会混乱。

Tool 接口

class Tool:
    """所有 Tool 的基类。只需要实现 execute() 即可接入框架。"""
    def execute(self, params: dict[str, Any]) -> ToolResult:
        raise NotImplementedError

@dataclass
class ToolResult:
    success: bool
    output: str = ""
    error: str = ""

框架对 Tool 没有任何其他要求——可以是 shell 命令、API 调用、本地函数。测试中使用 MockTool 来验证行为：

class MockToolA(Tool):
    def execute(self, params: dict) -> ToolResult:
        return ToolResult(success=True, output=f"tool_a result: {params}")

Skill 定义完整示例

一个"代码审查" Skill 的完整定义：

Skill(
    id="code-review",
    name="Code Review",
    metadata={
        "tags": ["code", "review", "quality"],
        "intent_class": "code_review",
    },
    examples=["review this PR for bugs"],
    tools=[ToolRef(name="check"), ToolRef(name="report")],
    pipeline=[
        ExecutionStep(tool="check", params={"mode": "quick"}),   # Step 1: 检查
        ExecutionStep(tool="report", params={"format": "md"}),    # Step 2: 报告
    ],
)

执行时：check(params={"mode": "quick"}) → report(params={"format": "md"})。

Skill Executor vs Orchestrator

维度SkillExecutorOrchestrator
职责执行单个 Skill 的 Tool 序列分解任务、调度子 Agent 执行多个 Skill
粒度单次执行全生命周期管理
错误处理失败即停，跳过后续评估结果，决定重试或跳过
并行串行（按 pipeline 顺序）支持 Fork (CONCURRENT) 模式

要记住的一句话：SkillExecutor 做的是"按剧本走一遍"——它不管剧本好不好（那是 Router 的事），也不管演砸了怎么办（那是 Orchestrator 的事）。只做一件事，做简单。
