Context Compressor：分层上下文压缩

Lesson 007 · 阅读时间 7 分钟

长会话是 Agent 框架最大的敌人之一——随着对话进行，上下文窗口被大量工具输出填满，关键信息被稀释，Agent 越来越「笨」。Context Compressor 就是为此设计的。

四阶段 pipeline

四个阶段逐级执行，后一阶段是前一阶段的兜底：

1
SummaryPreview
将大工具输出压缩为结构化摘要，保留关键信息（数字、标识符、key-value）
输出：截断文本 + 元数据摘要
↓ 如果内容仍过大
2
PlaceholderReplacement
将大内容替换为 {{placeholder:uuid}} 令牌，原文存入外部缓存
输出：占位符令牌 → 缓存映射
↓ 如果上下文需要
3
OnDemandRetrieval
从占位符恢复完整内容——只在你真的需要看的时候才展开
输出：原始内容（按需加载）
↓ 如果上下文超限
4
Fallback
当上下文超过硬限制时，裁剪到 N 条，优先保留含 error/fail/important 的关键项
输出：裁剪后的上下文子集

三种触发条件（OR）

🧠
Agent 自省
Agent 主动调用 signal_introspect()
🔢
轮数阈值
达到预设对话轮数
📊
Token 阈值
上下文使用率超过阈值

三者是 OR 关系——任意一个满足就触发压缩。触发后 reset_metrics() 清零计数器。

代码结构

class CompressorPipeline:
    def __init__(self, token_threshold=2000, turn_threshold=10):
        self.stage1 = Stage1SummaryPreview()
        self.stage2 = Stage2PlaceholderReplacement()
        self.stage3 = Stage3OnDemandRetrieval()
        self.stage4 = Stage4Fallback(max_items=50)

    # 触发接口
    def update_metrics(self, token_count, turn_count)    # 累积指标
    def signal_introspect(self)                           # 自省信号
    def should_trigger(self) -> bool                     # OR 检查
    def reset_metrics(self)                              # 清零

    # Pipeline 执行
    def compress(self, content) -> str                   # Stage 1 → 2
    def decompress(self, token) -> str | None            # Stage 3
    def prune(self, items) -> list                       # Stage 4

设计要点

缓存友好

占位符 {{placeholder:uuid}} 只有几十个字符，在上下文中几乎不占空间。Claude Code 的 Prompt Cache 能更好地命中——因为绝大部分上下文在压缩后是稳定的、重复出现的占位符，而非每次都不同的大段文本。

逐级执行

不是每次压缩都跑满四阶段。大多数场景：

  • Stage 1 就够了——输出不大，截断一下就能用
  • Stage 2 是主力——大内容替换为占位符，使用时才展开
  • Stage 3 是按需——只在需要查看详情时调用
  • Stage 4 是最后防线——上下文真的快满了才裁剪

Stage 2 vs Stage 3

为什么要两个占位符机制？

  • Stage 2：{{placeholder:key}} —— 用于明确的「内容替换」，框架内部自动管理
  • Stage 3：{{ph:key}} —— 用于「on-demand 检索」，开发者可以主动请求恢复

实际使用中 Stage 2 是主力，Stage 3 是扩展点。

要记住的一句话：压缩不是丢信息，是给信息分层——热数据留在上下文里，温数据用占位符代替，冷数据存到记忆系统。需要时按层级恢复，而不是一股脑全塞进去。
